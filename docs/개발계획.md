## 4-1. 개발 교훈 (2026-04-19 업데이트)

실제 시행착오에서 얻은 원칙. 신규 작업 시 이 문서 먼저 확인.

### UX / 프론트엔드
- **공통 로직은 반드시 헬퍼로 추출**: 마을/검색/핀 클릭 지도 이동이 3곳에 중복되어 있다가 줌 레벨/애니메이션 변경 시 각각 수정해야 했음. `moveMapTo` 하나로 통일.
- **애니메이션 시 좌표 밀림 방지**: 카카오맵 `setLevel` 직후 `panTo` 를 호출하면 레이아웃 미완료로 중심이 밀림. `setLevel` → `requestAnimationFrame` → `panTo` 순서 필수.
- **React Portal 로 부모 transform 탈출**: 사이드바에 있는 모달이 사이드바 너비에만 갇혔음. `createPortal(..., document.body)` 로 해결.
- **카카오 CustomOverlay 는 SDK anchor + CSS transform 이중 보정 금지**: `yAnchor:1` + `transform: translate(-50%, -100%)` 둘 다 적용하면 컨텐츠가 좌표보다 카드 높이만큼 더 위로 떠버림 → 좌표 기준으로 그려지는 펄스/내접원과 시각적으로 어긋남. SDK anchor 하나만 쓰고 CSS transform 은 제거 ([globals.css](../web/app/globals.css) `.kepco-card-marker`).
- **iOS Safari 날짜 picker**: `opacity-0` overlay, `showPicker()` API 등 여러 시도가 모두 실패. 결국 네이티브 `<input type="date">` 가 가장 호환성 좋음.
- **덕지덕지 금지**: 문제가 여러 번 나타나면 각각 패치하지 말고 근본 원인(타이밍/레이아웃/중복 로직)을 찾기.

### 비기술 사용자 대응 (태양광 사업자)
- "GitHub Actions", "워크플로우" 같은 기술 용어 금지. "자동으로 실행됨, 창 닫아도 계속 진행" 식으로 표현.
- 말투는 `~함 / ~가능 / ~권장` 단정형. `~요 / ~세요` 는 피함.
- 개발자용 설정(상세 설정 등)은 `NODE_ENV === "development"` 로 localhost 에서만 노출.
- 새로고침 같은 "부담 큰" 액션은 `window.confirm` 으로 한 번 막기 — 자동 갱신 중임을 안내.

### 데이터 / 성능
- 배포판에서 느린 건 대개 Vercel cold start + Supabase RPC 지연. 쿼리 최적화 전에 cold start 가 원인인지 먼저 확인.
- UNIQUE 제약은 자동으로 B-tree 인덱스 생성됨 (중복 인덱스 추가 금지).
- **수시 변경되는 데이터 API 는 CDN 캐시 금지**: `/api/map-summary` 를 `s-maxage=3600` 으로 뒀다가 수집/지오코딩 끝났는데도 8시간 묵은 응답이 HIT 되어 "마커 0" 증상 발생. 클라이언트 `cache: "no-store"` 만으로는 CDN 계층을 못 건드림 → 응답 헤더 자체를 `Cache-Control: no-store` 로 바꿔야 함.
- **PostgREST/Supabase JS 는 기본 1000행 silently 잘림**: `.select()` 만 쓰면 응답이 잘려도 에러 없이 row 수만 줄어듦. 전량 반환이 필요한 API 는 반드시 `.range(from, from+PAGE-1)` 페이지네이션 (`/api/map-summary` 사례).
- **2,000개 이상 카카오 마커는 화면 보이는 것만 그릴 것**: rows 전체로 `Marker × 2,678개 + SVG→PNG 변환` 하면 메인 스레드 16초+ 점유. 해결 패턴 ([KakaoMap.tsx](../web/components/map/KakaoMap.tsx)):
  1. `idle` 이벤트(팬/줌 종료)에 200ms debounce → 현재 `bounds.contain()` 안 rows 만 마커 생성
  2. bounds 동일 시 rebuild skip → `addMarkers()` 가 idle 재발화하는 자기 루프 차단
  3. rebuild 함수는 ref 에 저장, props 변경 시 ref 만 갱신 → React effect 재실행 없이 idle 에서 직접 호출
  4. 카드 디자인은 SVG→PNG 대신 **HTML CustomOverlay** (`.kepco-card-marker` 클래스, [globals.css](../web/app/globals.css)) — DPR 자동 대응 + 즉시 렌더
  5. 클러스터러는 1회 생성 후 `clear()/addMarkers()` 만 반복 (재생성 비쌈)
  6. 카드 클릭은 지도 컨테이너에 위임 1개 (`data-addr` 로 식별) — 마커마다 핸들러 부착 시 closure 꼬임 사례 있음
  - 결과: 초기 16초 → 1.3초, FPS 9 → 60
- **REFRESH MATERIALIZED VIEW 는 PostgREST 기본 8초 timeout 에 잘릴 수 있음**: 데이터 누적되면 `statement_timeout` 으로 500. RPC 함수에 `SET statement_timeout = '5min'` 으로 함수 호출 동안만 우회. 동시 호출/연타 방어는 메타 테이블 `last_refreshed_at` 으로 60초 cooldown + `pg_try_advisory_lock` (비차단) 2단계 — 재호출은 즉시 `skipped:true` 반환해 사용자엔 "완료" 로 보임. Next.js Route 도 `export const maxDuration = 300` 필요 ([018_refresh_lock_timeout.sql](../db/migrations/018_refresh_lock_timeout.sql)).
- **검증 코드 결함 가능성을 먼저 의심 (2026-04-24)**: 진단 SQL/스크립트가 0% 또는 100% 같은 극단치를 내면 데이터 문제로 단정 말고 코드부터 점검. 사례: ① bjd_master 매칭 진단 SQL 이 NULLIF('-기타지역') 을 한 필드(addr_gu)에만 적용해 "매칭률 0%" 오판 → 전 5필드 정규화 후 83% 정상. ② PNU 구성 검증 시 산 가설(0/1) 오류로 100% NO_FEATURE → 실제 응답 1건 직접 확인해 1/2 정정. **비교 양쪽에 동일 정규화** 필수 + **단일 케이스 raw 응답 직접 확인**으로 가설 검증.
- **샘플은 케이스별 균등 추출**: 단순 `LIMIT N` 은 INSERT 순서로 한 동네만 가져옴 (다양성 0). 패턴별 (산/일반/특수) 별도 쿼리 후 병합으로 다양성 확보. `ORDER BY id DESC + LIMIT large` 후 Python `random.sample` 도 효과적.

VWorld 지오코딩 단계 자체가 불필요해져 호출 1회 절약 + 정확성 ↑ (실측 매칭률 ~93%, 실패 = KEPCO 원본 데이터 오염/폐지 필지). ([test_pnu_construction.py](../crawler/test_pnu_construction.py))

### 아키텍처 / 네이밍 (2026-04-24)
- **Atomic API + 클라이언트 enrichment 패턴** ([memory: project_atomic_api_pattern.md](../.claude/memory/project_atomic_api_pattern.md)): kepco_capa 정규화 후 raw row 는 bjd_code + 시설/용량만. UI 가 기대하는 주소/좌표는 `bjd_code → MapSummaryRow` in-memory 매핑으로 채움 (DB 재조회/조인 없음). atomic endpoint 5종 (`/api/capa/by-bjd`, `/api/capa/by-jibun`, `/api/parcel/by-pnu`, `/api/parcel/by-latlng`, `/api/polygon/by-bjd`) 은 모두 raw 반환, 합성은 클라이언트 책임 — 미래 다른 데이터소스 (인허가/건축물대장/조례 등) 가 추가돼도 동일 패턴.
- **함수 네이밍 컨벤션 `[verb][Source][Entity][By+Input]`** ([memory: feedback_naming_convention.md](../.claude/memory/feedback_naming_convention.md)): `fetchKepcoCapaByBjdCode`, `fetchVworldParcelByPnu`, `enrichKepcoCapaRowsWithVillageInfo`, `buildPnuFromBjdAndJibun` 같이 이름만으로 입력/출처/엔티티/조회키 모두 식별. 길지만 미래 30+ 함수 추가 대비 충돌 0 + grep 정확성. handler 도 동일 정신 (`openVillagePanelOnMarkerClick`, `openParcelPanelOnJibunClick`).
- **client-side fetch wrapper 와 server-side lib 분리**: `lib/api/*.ts` (kepco/vworld) = 컴포넌트가 호출하는 atoms (fetch wrapper, source 별 분리). `lib/vworld/*.ts` = route.ts 가 직접 호출하는 server-only lib. 이름 (`api` vs `vworld`) 으로 책임 분리. 컴포넌트 안 fetch URL 인라인 절대 금지 — URL 변경 시 grep 한 곳만.

### 크롤링 / 장기 작업
- **예외를 조용히 삼키지 말 것**: `except Exception` 에서 로그만 찍고 끝내면 상위가 정상 완료로 오판함. 반드시 **플래그에 기록** 하거나 **재전파** 해서 호출자가 상태를 구분 가능해야 함. (Job #180 사례: 주소 목록 API 일시 실패를 `except` 가 삼킴 → `completed` 오판 → 군위군 1/9 만 처리된 채 종료, [CRAWLING.md §10](./CRAWLING.md))
- **체크포인트는 일관성 있게 저장할 것**: 상위 레벨(동/면) 진입 시 하위 인덱스(리/지번)를 리셋하지 않으면 부분 실패 시 체크포인트가 "우보면인데 리는 소보면 것" 같은 불일치 상태로 저장됨 → 재개 시 엉뚱한 영역을 스킵.
- **외부 API 는 간헐적으로 실패한다고 가정**: KEPCO API 는 새벽 시간대 배치/백업으로 추정되는 일시 장애 가능. 주소 목록 같은 "계층 탐색" API 는 세션 재생성 + 점진 대기 재시도 필수.
- **재시작 경로는 "미완료 = stopped" 로 통일**: 타임아웃·예외·과다에러 모두 `stopped` 로 분류해야 기존 `auto_continue` 체이닝이 한 경로로 처리. `cancelled` 는 **사용자 명시적 취소** 로만 예약.
