# 원칙

외부 검색 API 호출 시 **검색조건을 빈값으로 보내면 더 많이/전부 다 받을 것** 이라는 직관은 틀릴 수 있다.

사이트가 기본 정렬/필터를 자동 적용하는 경우, **조건 빈값 = 종결/오래된 매물 위주로 응답** 같은 변질이 발생한다.

# Why

**2026-05-04 법원경매 court 사이트 사례**:

`/api/auction/by-pnu` 가 court 호출 시 `bidBgngYmd/bidEndYmd` (매각기일) 를 빈값으로 보냈음.
직관: "조건 안 보내면 모든 매물 다 와야지"

실측:
| 조건 | totalCnt | 740-7 매물 |
|---|---|---|
| 매각기일 빈값, emdCd=101 | 8건 | ❌ 누락 |
| 매각기일 빈값, emdCd 빈값 | 29건 | ❌ 누락 |
| 매각기일 6개월, emdCd=101 | 12건 | ✅ 포함 |
| 매각기일 6개월, emdCd 빈값 | 42건 | ✅ 포함 |

→ court 사이트가 매각기일 조건 없으면 **종결 매물 위주로** 응답. 신건/진행 매물이 첫 50건에서 빠짐.

이 함정에 빠져 "court 사이트가 740-7 누락" → "BJD 매칭 문제" → "fallback 2단계 추가" 등 엉뚱한 방향으로 4~5시간 헤맴.

의뢰자 한 줄: "여기서는 잘 나왔잖아 뭐가 다른건데?" — 정답은 매각기일 조건 차이.

# How to apply

**외부 검색 API 호출 시**:
1. **사이트 검색 폼의 기본값을 확인** — 매각기일 6개월 기본, 신건 체크 기본 등
2. **빈값 보내기 전에** 사이트가 빈값을 어떻게 해석하는지 의심
3. 응답 totalCnt 가 의외로 적으면 → 검색조건이 아닌 **누락 가능성** 부터 검증
4. 외부 API 레퍼런스 명세보다 **실제 사이트 검색 폼의 hidden defaults** 가 진실

**디버깅 순서**:
- 같은 조건으로 사이트 검색 폼에서 직접 검색 → 응답 비교
- 검색조건 하나씩 추가/제거하며 totalCnt 변화 추적

// 매각기일 6개월 윈도 — court 가 이 조건 없으면 진행 매물 누락하는 결함 회피
