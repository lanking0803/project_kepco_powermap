# PDF 양식 확장 시 컬럼 정의를 양식 표시보다 우선

견적서/리포트 PDF 양식을 코드로 옮기면서 의뢰자가 요청한 추가 시나리오를
구현할 때, **원본 PDF 가 보여주는 표시 형태만 보고 양식 의도를 추측하면
안 된다**. 컬럼 정의(헤더 부제, 계산식)를 먼저 읽어야 한다.

**Why:**
2026-05-06 봉남리 견적서 수익분석 사고. 봉남리 PDF 는 자기자본 100% 케이스
(대출액 0원)였고, "평균 상환액(年)" 컬럼이 1~20년차 모두 `-` (대시) 였다.
의뢰자 요청으로 10/15/20년 대출 시나리오를 추가하면서, "PDF 양식이 매년
동일 값 표시 = 균등 분배 형식이다"로 오해해 분석기간 20년 전체로 평균
분배해버림. 결과: 10년 대출인데 12~20년차에도 상환액 표시되는 버그.

근데 컬럼 정의는 `(=대출액+이자)` 였음 — "그 해 납부할 대출액+이자" 라는
뜻이었고, 양식이 매년 동일했던 건 단순히 대출액이 0이라 전부 `-` 였을
뿐인데 그걸 "균등 분배"로 오독함.

심지어 잘못된 동작을 테스트로 굳혀놨음(`모든 연차 loanPayment 동일 (평균값)`).
의뢰자가 직접 견적 뽑아보고 지적할 때까지 발견 못 함.

**How to apply:**
- PDF 양식을 코드로 옮길 때, **헤더 부제/계산식 정의를 먼저 읽기**.
  컬럼 표시 형태(매년 같은지, 비어있는지)는 부수 정보일 뿐.
- 의뢰자가 추가 시나리오를 요청하면 **원본 PDF 에서 그 시나리오가 어떻게
  표현되는지 데이터가 없을 가능성** 을 의심. 데이터 없으면 의뢰자한테 직접
  표현 방식 컨펌받기.
- 봉남리 같은 "단일 케이스 PDF" 는 함정. 한 가지 시나리오만 보면 양식이
  지원하는 다른 케이스의 표현 의도를 알 수 없음.
- 새 시나리오 테스트는 **결과 검증** 으로 짜야지 **현재 동작 검증** 으로
  짜면 버그가 테스트로 굳어짐. 봉남리 검증값 같은 외부 출처 숫자가 있을 때만
  테스트 신뢰 가능.
- 변수 변경 시나리오(시나리오 토글/금리/거치 등)는 **눈으로 한 번** 견적
  뽑아보기. 단위 테스트 통과 ≠ 영업 자료 정확.
